# راهنمای استقرار DocEdit روی Vercel (رایگان)

این راهنما مخصوص اکانت گیت‌هاب **masood5400** نوشته شده. کل مسیر حدود ۱۰ دقیقه طول می‌کشد و همه‌چیز در پلن رایگان انجام می‌شود.

## مرحله ۱ — ساخت ریپازیتوری در گیت‌هاب

ساده‌ترین راه (بدون نیاز به نصب git):

۱. برو به <https://github.com/new>
۲. نام ریپو را بگذار `docedit` (یا هر نامی) و **Create repository** بزن.
۳. در صفحه ریپو روی «uploading an existing file» کلیک کن.
۴. همه فایل‌ها و پوشه‌های همین فولدر را با درگ‌اند‌دراپ آپلود کن (پوشه‌های `api`، `css`، `js` و فایل‌های ریشه). فایل `run-windows.bat` هم اشکالی ندارد باشد.
۵. **Commit changes** بزن.

> اگر git نصب داری، به‌جای آپلود دستی:
> ```
> cd "D:\Resource\document editor"
> git init
> git add .
> git commit -m "DocEdit"
> git branch -M main
> git remote add origin https://github.com/masood5400/docedit.git
> git push -u origin main
> ```

## مرحله ۲ — اتصال به Vercel

۱. برو به <https://vercel.com> و با گزینه **Continue with GitHub** (اکانت masood5400) وارد شو.
۲. **Add New → Project** و ریپوی `docedit` را **Import** کن.
۳. هیچ تنظیم Build لازم نیست (پروژه استاتیک + Serverless Functions است). فقط **Deploy** بزن.

## مرحله ۳ — اتصال دیتابیس Upstash Redis (رایگان)

۱. در داشبورد پروژه در Vercel، تب **Storage** را باز کن.
۲. **Create Database → Upstash → Redis** را انتخاب کن (پلن **Free**).
۳. Connect بزن — Vercel خودش متغیرهای محیطی (`UPSTASH_REDIS_REST_URL` و `UPSTASH_REDIS_REST_TOKEN` یا `KV_REST_API_URL` و `KV_REST_API_TOKEN`) را به پروژه اضافه می‌کند. کد ما هر دو نام را پشتیبانی می‌کند.

## مرحله ۴ — متغیرهای محیطی

۱. در پروژه Vercel: **Settings → Environment Variables**
۲. دو متغیر بساز:
   - `ADMIN_PASSWORD` → گذرواژه پنل مدیریت ارشد (فقط برای خودت — صفحه `admin.html`)
   - `SESSION_SECRET` → یک رشته تصادفی طولانی (برای امضای نشست کاربران؛ اگر نگذاری از ADMIN_PASSWORD مشتق می‌شود ولی جدا بودنش بهتر است)
۳. ⚠️ مهم: بعد از هر تغییر متغیرها، از تب **Deployments** روی آخرین deployment گزینه **Redeploy** بزن — متغیرها فقط روی deploy های جدید اعمال می‌شوند.

## مرحله ۵ — نحوه کار سیستم

### کاربران و مهمان‌ها
- همه می‌توانند **مهمان** کار کنند (ویرایش، خروجی، حتی انتشار). موقع اشتراک‌گذاری پیشنهاد ثبت‌نام داده می‌شود.
- **ثبت‌نام/ورود ترکیبی و سریع است**: نام کاربری + گذرواژه؛ اگر حساب نباشد همان لحظه ساخته می‌شود (۱۰ ثانیه).
- **کامنت‌گذاری فقط با ورود** ممکن است.
- کاربران واردشده: پوشه می‌سازند (`dashboard.html`)، سند در پوشه می‌گذارند، پوشه یا سند را با نام کاربریِ دقیق دیگران به اشتراک می‌گذارند (خواندن یا خواندن+ویرایش).

### لینک اختصاصی
- در ویرایشگر 🌐: تعداد مرورگر یونیک، اجازه کامنت، و «مهمان بدون حساب ببیند یا نه» را تعیین می‌کنی.
- **سازنده لینک هرگز در شمارش حساب نمی‌شود** و همیشه می‌تواند لینک خودش را باز کند.
- حداکثر ۱ = اولین بازکننده لینک را برای بقیه قفل می‌کند.
- کاربرانی که مستقیم دسترسی گرفته‌اند هم در شمارش حساب نمی‌شوند.

### کامنت سبک گوگل‌داکس
- خواننده متن را انتخاب می‌کند → دکمه 💬 → کامنت به همان تکه متن سنجاق می‌شود و در سند هایلایت می‌شود.
- مالک و نویسنده کامنت می‌توانند **پاسخ** بدهند، **✓ انجام شد** بزنند یا **نادیده** بگیرند (کمرنگ می‌شود).
- با @ می‌توان کاربرانِ دارای دسترسی را منشن کرد.

### پنل مدیر ارشد (`admin.html`)
با `ADMIN_PASSWORD` وارد می‌شوی و **همه چیز** را می‌بینی: همه کاربران، همه اسناد (حتی مهمان‌ها)، محتوای هر سند و کامنت‌هایش؛ و می‌توانی گذرواژه کاربران را بازنشانی کنی.

### امنیت
- گذرواژه‌ها با scrypt هش می‌شوند؛ نشست‌ها HMAC امضا می‌شوند.
- خطاهای یکنواخت و پاسخ‌های عمومی: از روی API نمی‌شود فهمید چه سندی یا چه کاربری وجود دارد (ضد شمارش/enumeration).
- محدودیت نرخ روی ورود، اشتراک‌گذاری و کامنت (ضد بروت‌فورس).
- شناسه‌ها تصادفی و غیرقابل حدس‌اند.

> ⚠️ نکته صادقانه: «مرورگر یونیک» یعنی شناسه ذخیره‌شده در مرورگر. حالت ناشناس یا پاک‌کردن حافظه، مرورگر جدید حساب می‌شود. برای مستندات معمولی کاملا کافی است، ولی قفل رمزنگاری‌شده نیست.

## اجرای محلی (مثل قبل)

`run-windows.bat` را اجرا کن. ویرایش و خروجی‌گرفتن کاملا آفلاین-محور است؛ فقط انتشار/اشتراک به سرور Vercel وصل می‌شود (آدرس سرور را در پنل 🌐 وارد کن).
